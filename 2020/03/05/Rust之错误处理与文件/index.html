Rust之错误处理和文件

错误处理

程序的运行过程中难免会遇见非正常情况，这些非正常情况一般分为错误(Error)和异常(Exception)。错误一般都是有环境出现问题，或者编写的代码出现语法错误等情况，程序运行发生这种情况没有什么好的处理办法，要么不做处理，要么修改源代码。而对于可能发生也可能不会发生的异常，可以采取多种手段处理。

断言

Rust的类型系统使得编译器可以进行类型检查以便提现错误，而有些错误编译器并不能检查出来，需要断言来保证程序的正确运行。Rust提供了六种常用的断言，它们都是宏：

  1. assert!，要求参数必须是值为true的布尔表达式
  2. assert_eq!，要求两个表达式的值相等
  3. assert_ne!，要求两个表达式的值不能相等
  4. debug_assert!，assert!的debug版本，效果一样，只能用于调试模式
  5. debug_assert_eq!，同上
  6. debug_assert_ne!，同上

断言内部使用了panic!宏，引发线程恐慌。

Option

Option是一个枚举体，一个可选项，用来表示那些可能有值，也可能没有值的情况。每一个Option<T>类型的值要么包含一个Some(T)，要么包含一个None。在使用此类型时常配合模式匹配，因此需要考虑到值为None的情况，所以也常用于错误处理。用Option<Box<T>>实现可空的指针

枚举Option的常用方法：

  • is_some和is_none用于判断Option是否不为空或者为空，注意：用None给Option初始化时需要指定类型
  • as_mut方法将&mut Option<T>转换成Option<&mut T>
  • Some(T)的unwrap方法会取出其中的值，但是可能会引发恐慌，但是None调用此方法则会引起恐慌，推荐使用模式匹配或者unwrap_or方法，此方法将会试图获取并返回Option类型的变量中的值，若为None，则返回参数默认值。unwrap_or_else方法与其类似，但是参数是一个闭包
  • map，将一个Option中的值应用于一个闭包，并将返回将结果变成一个新的Option

Result

Result<T,E>是专门为错误处理设计的类型，它有两个值，要么是Ok<T>，要么是Err<E>，其中Ok表示正常情况下的返回值，Err表示错误情况下的返回值，下面的例子是分别将字符串”1”和”s”转化成数字类型，分别输出了两种不同的结果：成功和异常

还可以可以通过手动引发panic来应对那些处理不了的情况，相当与C++和Java的手动抛出异常。通过第三方库也可以做错误处理，如failure。

文件

目录

关于文件和IO的库在std::io和std::fs里，DirBuilder用于创建目录，其recursive决定是否递归创建，当参数为true时，路径的目录不存在时，递归创建；默认不递归创建，此时目录以存在，则create方法会报错 ，is_dir方法用于判断一个对象是否是目录。

问号运算符

读取指定路径的目录下的所有文件，这里的?号运算符表示当该处出现错误，提前返回错误，但是用到main函数里有一个问题，main函数的返回值是( )，若直接在main函数里使用问号运算符编译不会通过，就需要改变面函数的返回值类型

文件

File是系统文件的实例，当其超出作用域时会自动关闭，不需要手动关闭文件。create函数以只写模式打开一个文件，当文件不存在时，创建哪个文件，write_all写入一个[u8]的切片；open方法打开一个文件，若文件不存在，则返回一个错误，read_to_string方法用于将文件内容读取到一个字符串里，是std::io模块里的函数。
